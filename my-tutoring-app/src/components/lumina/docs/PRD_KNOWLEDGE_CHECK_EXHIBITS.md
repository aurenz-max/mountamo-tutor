# PRD: Knowledge Check Insets — Rich Inline Content for Assessment

> **Status:** Draft
> **Created:** 2026-04-09
> **Domain:** Assessment (knowledge-check)
> **Dependencies:** None (standalone inset system)

---

## Problem Statement

Knowledge Check today is **text-flat**. All 8 problem types present questions as plain text strings with plain text options. The question stem is always a `string`. The options are always `string[]`. There's no way to embed a math expression, a graph, a data table, a poem, a code snippet, or any structured content inline within a question or its options.

This limits the cognitive ceiling. Real assessments present **rich stimulus material** alongside questions — not as a separate orchestrated experience (that's what DeepDive does), but as inline content embedded directly in the question context. A math question shows an equation rendered in KaTeX. A science question shows a chart. A literature question embeds a poem stanza. An economics question presents a data table. The content is *part of the question*, not a wrapper around it.

**The gap today:**
- "Solve for x: 3x² + 7x - 2 = 0" renders as flat text — no proper math typesetting
- "Based on the following data..." has no data to show — it's all verbal description
- "Read the excerpt below..." has no formatted excerpt — just more body text
- "Analyze the trend in Figure 1..." has no figure

**What we want:**
- A KaTeX-rendered equation block embedded in the question stem
- A chart (bar, line, pie) rendered inline between the question and the options
- A formatted passage or poem with proper typography sitting inside the problem card
- A data table the student must read to answer the question
- A code block with syntax highlighting for CS questions
- All of these as **insets** — lightweight inline content blocks that are part of the problem data, generated by Gemini alongside the question, and rendered directly inside the problem primitive's card.

---

## Design Principles

1. **Insets, not exhibits.** This is not a separate orchestration layer. An inset is a typed content block that lives inside `BaseProblemData` and renders inline within the problem component. No wrapper primitives, no pinned panels, no separate renderers routing to external components.

2. **Zero DeepDive coupling.** DeepDive is a narrative orchestration primitive with its own layout engine, block system, and composition templates. Insets are small, focused renderers that live entirely within the knowledge-check problem-primitives space. They share no code, no types, no imports with DeepDive. If both need a table someday, that's a future extraction — not a starting constraint.

3. **Content that earns its place.** Every inset type must enable a class of questions that plain text cannot express. KaTeX enables proper math notation. Charts enable data interpretation. Passages enable reading comprehension. If an inset type doesn't unlock new question territory, it doesn't ship.

4. **Gemini generates insets and questions together.** The inset data (equation, table rows, chart data points, passage text) and the questions that interrogate it are generated in a single prompt call. This ensures coherence — the question actually tests what the inset shows.

5. **Progressive rollout.** Start with the highest-impact inset types (KaTeX, data table, passage) and layer in more over time. Each type is independent — shipping chart insets doesn't block or depend on KaTeX insets.

## Inset Type System

### The `Inset` Union Type

```typescript
// knowledge-check inset types

type InsetType =
  | 'katex'          // Math expressions and equations
  | 'data-table'     // Structured rows/columns of data
  | 'passage'        // Formatted text block (prose, poem, excerpt, primary source)
  | 'chart'          // Bar, line, or pie chart
  | 'code'           // Syntax-highlighted code block
  | 'image'          // Static image with caption (diagram, photo, map)
  | 'number-line'    // Annotated number line segment
  | 'definition-box';// Term + definition + usage (vocabulary in context)

interface BaseInset {
  insetType: InsetType;
  label?: string;   // Optional label: "Figure 1", "Equation", "Passage A", "Table 1"
}
```

### Concrete Inset Types

#### KaTeX Inset
Renders mathematical expressions with proper typesetting. Supports both inline and display (block) mode.

```typescript
interface KatexInset extends BaseInset {
  insetType: 'katex';
  /** LaTeX expression string — rendered by KaTeX */
  expression: string;
  /** 'display' = centered block equation; 'inline' = flows within text */
  displayMode: 'display' | 'inline';
  /** Optional descriptive caption below the expression */
  caption?: string;
}
```

**Example use cases:**
- "Solve for x" with a properly rendered quadratic
- "Which equation represents the relationship shown?" with equations in both question and options
- Physics formulas with subscripts, superscripts, Greek letters, fractions
- Statistical distributions with proper notation: P(X ≤ μ + 2σ)

**Rendering:** Use `katex` npm package (or `react-katex`). Render inside a glass-card inset panel with subtle border. Display mode centers the expression; inline mode flows with surrounding text.

#### Data Table Inset
Structured tabular data the student must read and interpret.

```typescript
interface DataTableInset extends BaseInset {
  insetType: 'data-table';
  headers: string[];
  rows: string[][];
  caption?: string;
  /** Optional: highlight specific cells to draw attention (row, col indices) */
  highlightCells?: Array<{ row: number; col: number }>;
}
```

**Example use cases:**
- "Based on the data in Table 1, which year had the highest growth rate?"
- Science experiment results → "What conclusion can you draw?"
- Historical data comparison → "Which region experienced the largest change?"

**Rendering:** Clean table with Lumina glass styling. Header row with accent color. Alternating row tints for readability. Highlighted cells get a subtle glow border. Responsive: horizontal scroll on narrow viewports.

#### Passage Inset
Formatted text block — prose excerpt, poem, primary source, speech, letter, or any extended text that serves as stimulus material.

```typescript
interface PassageInset extends BaseInset {
  insetType: 'passage';
  /** The text content — can include line breaks for poetry */
  text: string;
  /** Formatting style controls typography */
  format: 'prose' | 'poem' | 'quote' | 'letter' | 'source';
  /** Attribution: author, title, date */
  attribution?: string;
  /** Optional line numbers (for poetry/source analysis) */
  showLineNumbers?: boolean;
}
```

**Example use cases:**
- Literature: poem stanza → "What literary device is used in line 3?"
- History: primary source excerpt → "What perspective does the author reveal?"
- Science: method description → "What is the independent variable?"
- ELA: persuasive passage → "Which claim does the author support with evidence?"

**Rendering:**
- `prose`: Standard paragraph text with slight indent, serif-like feel (via font weight/spacing), blockquote border
- `poem`: Preserved line breaks, optional line numbers, centered or left-aligned, italic styling
- `quote`: Large opening quotation mark, attribution below, indented
- `letter`: "Dear..." header styling, signature footer
- `source`: Monospace-adjacent styling, line numbers, document-like border

#### Chart Inset
Simple data visualization — bar chart, line chart, or pie chart rendered inline.

```typescript
interface ChartInset extends BaseInset {
  insetType: 'chart';
  chartType: 'bar' | 'line' | 'pie';
  title: string;
  xLabel?: string;
  yLabel?: string;
  data: Array<{
    label: string;
    value: number;
    /** For grouped/stacked bar charts */
    group?: string;
  }>;
}
```

**Example use cases:**
- "According to the chart, which sector contributes most to GDP?"
- "What trend does the line graph show between 2010 and 2020?"
- "True or False: More than half of the budget goes to education."

**Rendering:** Use recharts (already a common Next.js dependency) or a lightweight custom SVG renderer. Lumina dark theme: dark background, glowing accent lines/bars, white labels. Keep it simple — 3-8 data points max. The chart is stimulus, not a dashboard.

#### Code Inset
Syntax-highlighted code block for CS/programming questions.

```typescript
interface CodeInset extends BaseInset {
  insetType: 'code';
  /** The source code */
  code: string;
  /** Programming language for syntax highlighting */
  language: string;
  /** Optional line numbers */
  showLineNumbers?: boolean;
  /** Optional: highlight specific lines */
  highlightLines?: number[];
}
```

**Example use cases:**
- "What does this function return when called with input 5?"
- "Identify the bug in the following code."
- "Which line would cause an infinite loop?"

**Rendering:** Monospace font, dark theme syntax highlighting (use Prism.js or similar with a dark Lumina-compatible theme). Line numbers in gutter. Highlighted lines get a subtle accent background. Rounded glass card container.

#### Image Inset
Static image with caption — diagrams, photos, maps, artwork reproductions.

```typescript
interface ImageInset extends BaseInset {
  insetType: 'image';
  /** Base64-encoded image data or URL */
  src: string;
  altText: string;
  caption?: string;
  /** Max width constraint: 'small' (200px), 'medium' (400px), 'full' (100%) */
  size?: 'small' | 'medium' | 'full';
}
```

**Example use cases:**
- Biology: cell diagram → "Identify the labeled structure"
- Geography: map → "Which region is indicated by the arrow?"
- Art: painting → "What artistic movement does this represent?"

**Rendering:** Rounded container, caption below in muted text, responsive sizing. Images from Gemini generation arrive as base64.

#### Number Line Inset
Annotated number line segment for math questions.

```typescript
interface NumberLineInset extends BaseInset {
  insetType: 'number-line';
  /** Start and end of the visible range */
  min: number;
  max: number;
  /** Tick marks to show */
  ticks: number[];
  /** Named points plotted on the line */
  points?: Array<{ value: number; label: string; color?: string }>;
  /** Optional shaded region */
  region?: { from: number; to: number; label?: string };
}
```

**Example use cases:**
- "Which point on the number line represents -3/4?"
- "What is the distance between points A and B?"
- "Which inequality is represented by the shaded region?"

**Rendering:** Custom SVG. Horizontal line with tick marks, labeled points as colored dots above the line, optional shaded region with dashed boundaries. Compact — fits inside a problem card without dominating it.

#### Definition Box Inset
Term + definition + usage context for vocabulary-in-context questions.

```typescript
interface DefinitionBoxInset extends BaseInset {
  insetType: 'definition-box';
  term: string;
  definition: string;
  partOfSpeech?: string;
  exampleSentence?: string;
}
```

**Example use cases:**
- "Based on the definition, which sentence uses 'ambiguous' correctly?"
- "Which synonym best replaces the defined word in context?"

**Rendering:** Compact card with bold term, italic part of speech, definition text, and example sentence in a lighter tint. Dictionary-entry styling.

## Problem Data Extension

### Single Inset (Most Common)

Every problem type gains an optional `inset` field in `BaseProblemData`:

```typescript
interface BaseProblemData {
  id: string;
  difficulty: ProblemDifficulty;
  gradeLevel: string;
  rationale: string;
  teachingNote: string;
  successCriteria: string[];
  // NEW
  inset?: Inset;
}
```

The inset renders between the question label and the question text (or between question text and options, depending on the problem type — see rendering rules below).

### Multiple Insets (Advanced)

For questions that reference multiple pieces of content (e.g., "Compare Table 1 and Figure 2"):

```typescript
interface BaseProblemData {
  // ...existing fields...
  inset?: Inset;
  /** Additional insets for multi-stimulus questions */
  additionalInsets?: Inset[];
}
```

Phase 1 implements `inset` only. `additionalInsets` is Phase 3.

### KaTeX in Options

For math-heavy questions, options themselves may need KaTeX rendering. Rather than making every option a rich type, add an `optionFormat` hint:

```typescript
interface MultipleChoiceProblemData {
  // ...existing fields...
  /** When 'katex', option text strings are rendered through KaTeX */
  optionFormat?: 'text' | 'katex';
}
```

This keeps the option data as simple strings (good for Gemini generation) while enabling proper rendering.

## Rendering Rules

### Where Insets Appear

Each problem type renders the inset in a consistent position:

| Problem Type | Inset Position |
|--------------|---------------|
| Multiple Choice | Between question text and options grid |
| True/False | Between statement text and True/False buttons |
| Fill in Blanks | Above the text-with-blanks area |
| Matching | Above the two-column matching area |
| Sequencing | Above the sortable item list |
| Categorization | Above the categorization workspace |
| Short Answer | Between question text and textarea |
| Scenario | Between scenario card and question text |

### The `InsetRenderer` Component

A single component routes inset data to the correct renderer:

```typescript
// primitives/problem-primitives/InsetRenderer.tsx

interface InsetRendererProps {
  inset: Inset;
  className?: string;
}

const InsetRenderer: React.FC<InsetRendererProps> = ({ inset, className }) => {
  switch (inset.insetType) {
    case 'katex':          return <KatexInsetRenderer data={inset} />;
    case 'data-table':     return <DataTableInsetRenderer data={inset} />;
    case 'passage':        return <PassageInsetRenderer data={inset} />;
    case 'chart':          return <ChartInsetRenderer data={inset} />;
    case 'code':           return <CodeInsetRenderer data={inset} />;
    case 'image':          return <ImageInsetRenderer data={inset} />;
    case 'number-line':    return <NumberLineInsetRenderer data={inset} />;
    case 'definition-box': return <DefinitionBoxInsetRenderer data={inset} />;
  }
};
```

Each renderer is a small, self-contained component (50-150 lines) with no external primitive dependencies. They live alongside the problem primitives:

```
primitives/problem-primitives/
├── insets/
│   ├── InsetRenderer.tsx
│   ├── KatexInsetRenderer.tsx
│   ├── DataTableInsetRenderer.tsx
│   ├── PassageInsetRenderer.tsx
│   ├── ChartInsetRenderer.tsx
│   ├── CodeInsetRenderer.tsx
│   ├── ImageInsetRenderer.tsx
│   ├── NumberLineInsetRenderer.tsx
│   └── DefinitionBoxInsetRenderer.tsx
├── MultipleChoiceProblem.tsx
├── TrueFalseProblem.tsx
└── ...
```

### Styling Contract

All inset renderers follow the same visual contract:
- **Container:** `rounded-lg border border-white/10 bg-white/5 p-4` (subtle glass inset)
- **Label:** If `label` is set, render as small uppercase monospace text above the content: `text-xs font-mono uppercase tracking-wider text-slate-500`
- **Spacing:** `my-4` margin to separate from question text and options
- **Max height:** Insets that could grow tall (passage, code, table) get `max-h-64 overflow-y-auto` with a subtle fade gradient at the bottom when overflowing
- **Responsive:** All insets must work at mobile widths (320px+)

## Item Sets

Item sets group 2-4 questions around a shared inset. The inset is defined once and displayed persistently while questions paginate below it.

### Data Model

```typescript
interface ItemSetData {
  type: 'item_set';
  id: string;
  /** The shared inset rendered above all questions */
  sharedInset: Inset;
  /** 2-4 problems that reference the shared inset */
  problems: ProblemData[];
  gradeLevel: string;
}
```

### Layout

**Mobile / narrow:** Inset at top, questions below with pagination dots. Inset scrolls away naturally.

**Desktop (≥ 1024px):** Two-column layout. Inset sticky on the left (~40% width), questions scroll on the right (~60% width). This mirrors AP/SAT split-screen format.

### Bloom's Ladder

Item set generators produce questions at ascending Bloom's tiers against the same stimulus:

| Question | Tier | Pattern |
|----------|------|---------|
| Q1 | Recall | "What value does the table show for X?" |
| Q2 | Apply | "Using the data, calculate the change between Y and Z." |
| Q3 | Analyze | "What trend does the data reveal, and what might explain it?" |

### Evaluation

Each question in an item set evaluates independently. They share a common `exhibitId` for backend analytics — enabling insights like "students who read data correctly but can't analyze trends."

## Gemini Generation Strategy

### Schema Design

Each inset type gets its own focused generator function. The generator returns the inset data + problem data together to ensure coherence.

```typescript
// service/knowledge-check/gemini-knowledge-check-insets.ts

/** Generate a problem with a KaTeX inset */
generateKatexProblem(topic, gradeLevel, problemType, bloomsTier?):
  → { inset: KatexInset, problem: ProblemData }

/** Generate a problem with a data table inset */
generateDataTableProblem(topic, gradeLevel, problemType, bloomsTier?):
  → { inset: DataTableInset, problem: ProblemData }

/** Generate a problem with a passage inset */
generatePassageProblem(topic, gradeLevel, problemType, passageFormat, bloomsTier?):
  → { inset: PassageInset, problem: ProblemData }

/** Generate a problem with a chart inset */
generateChartProblem(topic, gradeLevel, problemType, chartType, bloomsTier?):
  → { inset: ChartInset, problem: ProblemData }

/** Generate an item set: shared inset + N problems at ascending Bloom's tiers */
generateItemSet(topic, gradeLevel, insetType, problemType, count, bloomsLadder?):
  → ItemSetData
```

### Prompt Rules

Every inset generator prompt must include:
1. **"The question must be unanswerable without the inset."** — This is the quality gate. If a student can answer the question from the stem alone, the inset is decorative and the generation failed.
2. **"Distractors must be derived from the inset content."** — For MC/T-F: wrong answers should be plausible misreadings of the data/passage/equation, not random alternatives.
3. **Schema simplicity.** Each generator sends a focused schema for its inset type (3-5 fields) + the standard problem schema. No mega-union schemas.

### KaTeX in Options

When `optionFormat: 'katex'` is set, the generator produces LaTeX strings for each option:

```json
{
  "question": "Which expression is equivalent to the one shown above?",
  "inset": { "insetType": "katex", "expression": "\\frac{d}{dx}[x^3 + 2x]", "displayMode": "display" },
  "options": [
    { "id": "a", "text": "3x^2 + 2" },
    { "id": "b", "text": "3x^2 + 2x" },
    { "id": "c", "text": "x^3 + 2" },
    { "id": "d", "text": "3x + 2" }
  ],
  "optionFormat": "katex",
  "correctOptionId": "a"
}
```

### New Controls

| Control | Type | Options | Purpose |

| **KaTeX Options** | Checkbox (when inset = katex) | — | Render options as LaTeX too |

### Generation Flow

```
User selects:
  Topic: "Derivatives"
  Inset Mode: "Single Inset"
  Inset Type: "katex"
  KaTeX Options: ✓
  Problem Type: "multiple_choice"
  Bloom's Tier: "Apply"
  Grade Level: "high-school"

→ Calls generateKatexProblem("Derivatives", "high-school", "multiple_choice", "apply")
→ Returns: { inset: KatexInset, problem: MultipleChoiceProblemData }
→ Merges: problem.inset = inset, problem.optionFormat = 'katex'
→ Renders: KnowledgeCheck with KaTeX equation above MC options (also KaTeX-rendered)
```

### Preview Features

1. **Inset preview** — The generated inset renders in full above the question, using the same `InsetRenderer` the student sees. Visual QA happens here.
2. **Raw data toggle** — Button to show the raw inset JSON for debugging Gemini output quality.
3. **Regenerate inset only** — Keep the question, regenerate just the inset data. Useful when the question is good but the stimulus data is weak.
4. **Regenerate question only** — Keep the inset, regenerate the question. Useful when the data is rich but the question doesn't interrogate it well.

## Eval Mode Implications

Insets raise the cognitive floor of existing eval modes. A `recall` question with a data table inset requires data literacy on top of content knowledge.

**Approach:** Add inset-aware eval mode variants as separate IDs:

| Eval Mode | Base Beta | With-Inset Beta | What Changes |
|-----------|----------|----------------|--------------|
| `recall` | 1.5 | — | No change — recall questions rarely need insets |
| `recall-exhibit` | — | 2.0 | Recall against presented stimulus |
| `apply` | 3.0 | — | No change |
| `apply-exhibit` | — | 3.5 | Apply knowledge to interpret stimulus |
| `analyze` | 4.5 | — | No change |
| `analyze-exhibit` | — | 5.0 | Multi-step analysis of stimulus |
| `evaluate` | 6.0 | — | No change |
| `evaluate-exhibit` | — | 6.5 | Judge claims against presented evidence |

The `-exhibit` variants use the same problem types but require an inset to be present. The generator selects inset type based on domain and Bloom's tier.

## What NOT To Do

1. **Don't import DeepDive components.** Build self-contained inset renderers. They're small (50-150 lines each) and the maintenance cost of coupling to DeepDive's block system far exceeds the cost of writing a simple table or passage renderer.

2. **Don't make insets decorative.** If the question is answerable without looking at the inset, the generation failed. The generator prompt must enforce: "Question must be unanswerable without the inset data."

3. **Don't create a mega Gemini schema.** Each inset type gets its own generator function. A union schema with all 8 inset types will produce malformed JSON. Keep schemas focused: 3-5 fields per inset type.

4. **Don't over-engineer interactivity.** Insets are display-only stimulus material. The student reads/interprets the inset and then interacts with the problem (MC options, drag-drop, text input). The inset itself has no interactive state. Exception: diagram explore-mode with clickable labels could come later, but is not in scope for Waves 1-3.

5. **Don't skip the tester.** Every inset type must be generatable and previewable in `KnowledgeCheckTester` before shipping. The tester is how we validate that (a) Gemini produces well-formed inset data and (b) the renderer handles it gracefully.

6. **Don't change the existing plain-text path.** `inset` is optional. All existing knowledge-check problems continue to work exactly as they do today. Zero regression risk.
